# 関係図整理ツール 実装指示書 for VS Code + Codex

## 目的

この文書は、VS Code 内の Codex に対して、**単一 HTML ファイルで動作する「関係図整理ツール」**を高い再現性で実装させるための実装指示書です。

対象ツールは、ただの関係図エディタではなく、以下を**非エンジニアでも編集・閲覧しやすい形で見える化する HTML ツール**です。

- 公開ページ
- 管理ページ
- DB接続
- ページ間導線
- CSS / JS 部品
- モーダル / ポップアップ
- AI相談 / AIひな形生成 / 完成品AI出力

主役は作図機能ではなく、**関係をわかりやすく整理する UI** です。

---

## Codex への最上位指示

Codex には、以下の方針を厳守させてください。

- 出力は **完成した単一 HTML ファイル 1つ** のみ
- CSS / JavaScript はすべて HTML 内に内包すること
- 外部ライブラリ、CDN、npm、ビルド工程は禁止
- オフラインでローカル動作すること
- 日本語 UI であること
- デスクトップ利用を主対象にすること
- 見た目は非エンジニア向けで、やさしい SaaS 風 UI にすること
- デモではなく、実用できる編集 UI にすること
- JSON 保存 / 読み込み / AI用プロンプト生成まで含めること
- 途中段階の STEP1 / STEP2 でも AI 相談と AI ひな形生成を使えること
- 完成系の AI 出力は STEP3 を主想定とするが、全段階で使えるようにしてよい

---

# 1. 想定ユーザー

優先順位は以下です。

1. AI使用を前提とした非エンジニアの企画者・依頼者
2. AI使用を前提とした、少しだけ HTML / CSS / JS を理解している人

そのため、UI 方針は以下にします。

- いきなり全部見せない
- 段階的に機能を解放する
- 主表示はページ名
- ファイル名は補助表示
- 詳細は右フォームで編集
- 初期表示は「かんたん表示」
- 詳細表示では html / CSS / JS / DB項目まで見える
- 途中段階でも AI に相談できる
- 途中段階でも AI にひな形を作ってもらえる

---

# 2. 画面構成

本ツールは、**単一 HTML ファイル内 SPA** として実装します。
画面遷移はページ再読み込みではなく、状態切り替えで実現します。

実装すべき画面は次の 4 つです。

1. TOPページ
2. STEP1編集画面（公開ページ整理）
3. STEP2編集画面（管理ページ追加）
4. STEP3編集画面（DB接続整理）

# 3. TOPページ仕様

## 役割

入口ページです。ユーザーがどの段階から始めるかを選択します。

## 必須UI部品

- アプリタイトル
- アプリ説明文
- 3つの開始カード
- 保存データ読込導線
- サンプル開始導線
- ヘルプ導線
- 進め方の流れ表示

## 必須文言

### ヘッダー
- タイトル: `関係図整理ツール`
- サブ説明: `非エンジニアでも、ページ構成・導線・DBの流れを見える化`

### メイン見出し
- `どの段階から始めますか？`

### 説明文
- `いきなり全部を作るのではなく、公開ページ → 管理ページ → DB接続 の順に整理できます。`

### 開始カード1
- 見出し: `① 最初から始める`
- 説明:
  - `公開ページのサイトマップと、`
  - `各ページのCSS / JSのつながりを整理します。`
- ラベル例:
  - `公開ページ`
  - `CSS / JS`
  - `ページ導線`
- ボタン:
  - `この段階から始める`

### 開始カード2
- 見出し: `② 公開ページのサイトマップは完成している`
- 説明:
  - `公開ページを前提に、`
  - `管理ページ側のサイトマップと各ページのCSS / JSのつながりを整理します。`
- ラベル例:
  - `公開ページ`
  - `管理ページ`
  - `CSS / JS`
- ボタン:
  - `この段階から始める`

### 開始カード3
- 見出し: `③ 公開ページ / 管理ページのサイトマップは完成している`
- 説明:
  - `公開ページと管理ページを前提に、`
  - `DB側との保存 / 参照のつながりを整理します。`
- ラベル例:
  - `公開ページ`
  - `管理ページ`
  - `DB接続`
- ボタン:
  - `この段階から始める`

### 補助メニュー
- `保存データを読み込む`
- `サンプルから始める`
- `ヘルプ / 使い方`

### 進め方の流れ
- `STEP1: 公開ページ整理`
- `STEP2: 管理ページ追加`
- `STEP3: DB接続整理`

# 4. 編集画面 共通レイアウト

STEP1 / STEP2 / STEP3 は、基本骨格を共通にします。

## 共通レイアウト

- 上部バー
- 左サイド（追加）
- 中央キャンバス
- 右サイド（詳細編集）
- 下部または上部の補助表示（段階、ヒント、凡例など）

## 上部バー

必須要素:
- 新規作成
- 保存
- 読み込み
- 書き出し
- 表示切替（かんたん表示 / 詳細表示）
- ヘルプ

STEP3 では加えて:
- AIに相談
- AIにひな形を作ってもらう
- 完成品をAIに出力

STEP1 / STEP2 でも、AI相談・AIひな形生成は使えるようにすること。
配置は右サイド内でもよいです。

## 左サイド

役割は「追加」。
ボタンは大きく、種別が分かること。

## 右サイド

選択中要素の詳細編集フォーム。
選択していない場合はガイド文を表示します。

## 中央キャンバス

3エリアを初期表示します。
エリアは半固定レイアウトです。

- 左上: `①公開ページエリア`
- 右上: `②管理ページエリア`
- 下部全幅: `③DB接続エリア`

エリア色分け:
- 公開ページエリア: オレンジ系
- 管理ページエリア: ブルー系
- DB接続エリア: グリーン系

中央キャンバスは表示領域よりも内部サイズをかなり広く取り、以下を可能にします。

- パン
- ズーム
- 100%表示
- 全体整列

エリア分割ラインはドラッグでサイズ調整可能にします。
ただし完全自由レイアウトにはしません。

# 8. 要素定義

## 8.1 エリア

3種類:
- `publicArea`
- `adminArea`
- `dbArea`

各エリアの保持項目:
- `id`
- `kind`
- `x`
- `y`
- `width`
- `height`
- `title`
- `color`

## 8.2 親箱

親箱は 1ページ / 1モーダル / 1ポップアップ を表します。

種類:
- `publicPage`
- `adminPage`
- `modal`
- `popup`

保持項目:
- `id`
- `kind`
- `areaKind`
- `linkedParentId`
- `name`
- `htmlFile`
- `title`
- `description`
- `notes`
- `x`
- `y`
- `width`
- `height`
- `collapsed`
- `cssItems`
- `jsItems`
- `linkHints`
- `dbHints`
- `stageVisibleFrom`

### linkedParentId の扱い
モーダル / ポップアップでは必須に近い扱いにします。
ただし作成時点では未接続でもよく、未接続時は警告表示を出します。
完成扱い・AI完成出力時には強い警告を出します。

## 8.3 html箱

親箱上段の本体情報ブロックです。

表示項目:
- html名
- タイトル
- 内容
- リンク導線

リンク導線は:
- 文字でも見える
- 接続線でも見える

## 8.4 CSS / JS 子箱

種別:
- CSS（見た目）
- JS（動き）

保持項目:
- `id`
- `kind`
- `fileName`
- `label`
- `role`
- `scope`

scope:
- `common`
- `specific`

初期表示は折りたたみ。
詳細表示で展開可能にします。

未設定時の注意書き例:
- CSS未登録
- JS未登録
- インライン記述の可能性あり

## 8.5 DB項目

DB項目は node として持ってよいです。

種別:
- `saveData`
- `referenceData`

保持項目:
- `id`
- `kind`
- `name`
- `tableName`
- `description`
- `x`
- `y`
- `width`
- `height`

## 8.6 接続線

最低3種類:
- `navigate`
- `save`
- `reference`

保持項目:
- `id`
- `fromId`
- `toId`
- `relationType`
- `label`
- `description`

### 許可ルール

#### navigate
- publicPage → publicPage
- publicPage → modal
- adminPage → adminPage
- adminPage → modal
- modal → page
- popup → page
- 必要に応じて page ↔ popup も可

#### save
- publicPage / adminPage / modal / popup → saveData

#### reference
- referenceData → publicPage / adminPage / modal / popup

### 原則禁止
- CSS / JS から DB への接続
- DB項目どうしの navigate
- saveData から page への save
- page から referenceData への reference の向き
- 無意味な自己接続

禁止接続はブロックするか、わかりやすい警告を出します。

# 12. AI連携の考え方

このアプリ自体は AI API を直接叩きません。
代わりに次を行います。

- AIに投げるためのプロンプトテキストを生成
- コピー
- `.txt` で保存

つまり AI連携とは:
**現在の構造状態を、外部 AI に理解しやすいテキストへ整形して出力する機能**
です。

各段階で使える AI 機能:
- AIに相談する
- AIにひな形を作ってもらう
- 完成品をAIに出力する

完成品出力は主に STEP3 を想定しますが、全段階で使えてもよいです。

# 13. AI用の状態シリアライズ仕様

AIに渡すテキストは、人間向けよりも AI が誤読しにくい構造化テキストを優先します。
アプリ内に、現在状態をテキスト化する関数を持たせてください。

## 出力フォーマット例

```text
[APP]
name: 関係図整理ツール
stage: STEP1
screen: editor
project_name: 〇〇
user_level: nonEngineer
view_mode: simple

[AREAS]
- publicArea: ...
- adminArea: ...
- dbArea: ...

[NODES]
- id: node_1
  kind: publicPage
  name: トップページ
  html_file: index.html
  title: トップページ
  description: 会社紹介・導線
  linked_parent_id: null
  css_items:
    - common.css | 共通CSS | ページ全体 | common
    - index.css | 個別CSS | トップ専用 | specific
  js_items:
    - common.js | 共通JS | 共通動作 | common
    - index.js | 個別JS | トップ専用 | specific
  links:
    - 一覧へ -> node_2
    - 問い合わせ -> node_4

[EDGES]
- edge_1 | navigate | node_1 -> node_2 | 一覧へ
- edge_2 | navigate | node_1 -> node_4 | 問い合わせ
- edge_3 | save | node_4 -> db_save_1 | 保存

[WARNINGS]
- モーダル node_4 は linkedParentId が未設定
- CSS未登録のノードあり: node_8

[USER_REQUEST]
...
```

この構造化テキストを、各 AI プロンプトの中に差し込みます。

- 管理ページが STEP1 で追加されようとした

- 厳しすぎるブロックはしない
