# 関係図整理ツール 差分修正指示書 for VS Code + Codex

## 位置づけ

この文書は、既存の **「関係図整理ツール 実装指示書 for VS Code + Codex」** に対する **差分修正指示** です。  
既存仕様は原則維持しつつ、今回のユーザーフィードバックを受けて、**UI/UX・ステップ導線・操作性・初回体験**を改善してください。

### 最重要方針

今回の修正では、次の方針を最優先してください。

* **「完成済みの図を眺めるツール」ではなく、「途中からでも作り始めやすい制作ツール」に寄せる**
* **STEP制の意味を崩さない**
* **初見ユーザーが迷わない導線を作る**
* **既存の良い見た目は壊しすぎず、操作性と理解しやすさを優先する**

\---

# 1\. 最優先で直すこと（P0）

## 1-1. STEP開始時の初期状態を全面的に見直すこと

現状の問題:

* STEP1 を選んだ時点で公開ページの箱がすでに置かれている
* STEP2 を選んだ時点で管理ページが置かれている
* STEP3 を選んだ時点で DB 項目まで置かれている
* サンプルも完成済みに近い状態で始まってしまう

これは、初心者にとって

* 何をしたらよいか分からない
* 既存の箱を壊してはいけない気がする
* STEPの意味がわからない
という混乱を生む

### 修正要件

#### STEP1「最初から始める」

* **完全白紙**で開始すること
* ただし公開ページエリア①だけは大きく表示し、何を追加すべきかのガイドは出してよい
* 最初に置かれているノードはゼロにすること

#### STEP2「公開ページのサイトマップは完成している」

* **STEP1 を最低限完了した状態のデータ**を読み込んで開始すること
* ここでの初期ノードは「公開ページのみ」
* 管理ページは未作成または最小限にして、ユーザーが管理ページ追加の作業を始められるようにすること
* 管理ページがすでに大量に置かれている状態で開始してはいけない

#### STEP3「公開ページ / 管理ページのサイトマップは完成している」

* **STEP2 を最低限完了した状態のデータ**を読み込んで開始すること
* ここでの初期ノードは「公開ページ＋管理ページ」
* DB 項目や保存 / 参照接続は未作成または最小限にし、ユーザーが DB を整理する余地を残すこと

#### サンプル開始

サンプルは「全部完成済み」ではなく、**段階別サンプル**に分けること。

最低限、各サンプルに対して次を持つこと:

* STEP1版サンプル
* STEP2版サンプル
* STEP3版サンプル

例:

* コーポレートサイト / STEP1版
* コーポレートサイト / STEP2版
* コーポレートサイト / STEP3版

\---

## 1-2. 各STEPでキャンバスのエリア比率を変えること

現状の問題:

* STEP1 でも ①公開 / ②管理 / ③DB がほぼ対等に見える
* その結果、STEP1 の主役が分かりにくい
* 公開ページエリアからノードがはみ出しやすい

### 修正要件

#### STEP1

* ①公開ページエリアを最大化すること
* ②管理ページエリアは小さく未開放表示
* ③DB接続エリアも小さく未開放表示
* STEP1では「①が主役」であることが見た目だけで分かるようにすること

#### STEP2

* ①公開ページエリアと ②管理ページエリアを主役にすること
* ③DB接続エリアは小さく未開放表示にすること

#### STEP3

* ①②③ を十分に使えるサイズで表示すること

### 追加要件

* ノード初期配置は必ず所属エリア内に収めること
* エリア外にはみ出したノードが初期状態で見えることを禁止する
* 「自動整列」時もエリア内に収めること

\---

## 1-3. STEP遷移導線とTOP導線を明示すること

現状の問題:

* TOPページへどう戻るか分かりにくい
* 次のSTEPにどう進むか分かりにくい
* 前STEPに戻る導線も弱い

### 修正要件

各編集画面の上部バーまたは右上に、常時見える導線を追加すること。

必須:

* `TOPへ戻る`
* `前のSTEPへ`
* `次のSTEPへ`

### 次のSTEPへ進む時の仕様

未保存変更や不足がある場合は、やさしい確認ダイアログを出すこと。

確認文の例:

* このまま次のSTEPに進みますか？
* 未保存の変更があります
* 不足項目がありますが、このまま進むこともできます

\---

## 1-4. 「保存」と「書き出し」の意味を明確に分けること

現状の問題:

* 非エンジニアには「保存」と「書き出し」の違いが分かりにくい

### 文言変更要件

上部バーの表記を以下へ変更すること。

* `保存` → `作業データを保存`
* `読み込み` → `作業データを開く`
* `書き出し` → `外部用に出力`

### 「外部用に出力」の内訳

メニューまたはサブメニューで、最低限次を出せること。

* JSONで出力
* AI相談用テキストを出力
* AIひな形用プロンプトを出力
* 完成品プロンプトを出力

\---

## 1-5. 離脱警告を入れること

現状の問題:

* 編集途中でページを離れても警告がない

### 修正要件

#### ブラウザ離脱時

未保存変更がある場合、`beforeunload` 相当の警告を出すこと

#### アプリ内遷移時

次の操作でも確認ダイアログを出すこと

* TOPへ戻る
* 別STEPへ移動
* サンプルを読み込む
* 新規作成する
* 別JSONを開く

\---

# 2\. UI/UXの重要修正（P1）

## 2-1. TOPページのレスポンシブ改善

現状の問題:

* width が狭いとTOPページが破綻し、ヘッダー以外が実質操作不能になる

### 修正要件

TOPページはスマホ幅でも最低限使えるようにすること。

最低条件:

* hero を1カラムに落とす
* 3つの開始カードを縦積みにする
* 補助説明・進め方も縦積みにする
* ボタンが画面外へ出ない
* 少なくとも「STEP選択」はスマホ幅で可能にすること

### 補足

編集画面の完全モバイル対応は不要  
ただし編集画面でも、狭い画面時に

* `この画面はPC表示推奨です`
のガイドを出せるとなおよい

\---

## 2-2. TOPページのヒーローヴィジュアルを差し替えること

現状の問題:

* 右上の `hero-visual` は悪くないが、説明的すぎて印象が弱い
* 「頑張って作った図」感がある一方、プロダクトの魅力訴求としては弱い

### 修正要件

既存の `hero-visual` は廃止し、**ヒーローヴィジュアル用の画像**を３枚並べて表示する方式にすること。

要件:

* imageフォルダ内の各画像（hero_STEP1_image.png及びSTEP2、STEP3）を左から並べる
* ブラウザサイズに対しておかしな表示にならないように留意する

### 実装要件

* 画像差し替えしやすいように、`img` タグまたは背景画像の差し替えやすい構造にすること
* `hero-visual` を CSS図形で無理に維持しないこと
* 画像ファイルが存在しない場合のフォールバック表示は残してよい

\---

## 2-3. フッター文言を修正すること

現状の問題:

* `ローカル単体HTML` 的な説明はTOPページのフッターには不要
* コピーライトの方が自然

### 修正要件

TOPページフッターには最低限次を入れること。

* `©2026 Yobibit`

必要なら右側に:

* 利用規約
* プライバシーポリシー

を置いてよい

\---

## 2-4. STEP1のワークスペースサイズを広げること

現状の問題:

* `workspace` が狭く感じる
* 初回から窮屈
* ノードを動かす余地が少ない

### 修正要件

* STEP1 のワークスペース内部サイズを十分に広げること
* 少なくとも初期ノード配置やドラッグ移動で窮屈に感じない大きさにすること
* パン/ズーム前提なら、体感として「狭い」と感じない広さにすること

\---

# 3\. 右パネル（ページ詳細）の修正（P1）

## 3-1. ファイル名入力と拡張子UIを改善すること

現状の問題:

* html/css/js のファイル名入力で、毎回拡張子まで書かせるとノイズになる

### 修正要件

#### ページ詳細

* htmlファイル名入力欄では、入力部とは別に右側へ `.html` を固定表示できるようにすること
* ただし `admin/edit` のようなパス入力は許可すること
* パス本体と拡張子の区切りが分かるUIにすること

#### CSS一覧 / JS一覧

* 必要に応じてベース名と拡張子を分けてもよい
* 少なくとも、拡張子の扱いがユーザーにとって分かりやすいUIにすること

\---

## 3-2. CSS一覧 / JS一覧の表示を改善すること

現状の問題:

* `|` 区切り表示は違和感がある
* 視認性が低い

### 修正要件

* CSS一覧 / JS一覧は **チップ表示** または **1行1項目** にすること
* 個別に削除 / 編集 / 並べ替えがしやすい見た目にすること
* 少なくとも視覚的に「項目のまとまり」が分かるようにすること

\---

## 3-3. AIタブ切替時にスクロール位置を保持すること

現状の問題:

* `ai-tabs` 内のボタンを押すとスクロールが上に戻る

### 修正要件

* AIタブ切替時に、右パネル全体のスクロール位置を不必要にリセットしないこと
* パネル全再描画を避け、AI領域だけ差し替えること
* もしくはスクロール位置を保存 / 復元すること

\---

## 3-4. リンク導線一覧は自動生成寄りにすること

現状の問題:

* リンク導線一覧を手入力させるより、接続線から自動反映した方が自然

### 修正要件

* リンク導線一覧は、`navigate` 接続線から自動生成すること
* ラベルは編集可能にしてよい
* 補足メモだけ別管理でもよい
* 図上の接続と右パネルの一覧がズレないことを優先すること

\---

# 4\. 左パネル（追加パネル）と操作性の修正（P1）

## 4-1. 接続線追加モードを一時モード化すること

現状の問題:

* 線接続モードに入ると箱移動ができなくなる
* 操作が重い
* モードから抜けにくい

### 修正要件

* 接続線追加は **一時モード** とすること
* 1本接続を作ったら自動で選択モードへ戻ること
* Escape で接続モード解除できること
* 必要なら Shift 押下で連続接続モードでもよい
* 箱移動と接続操作が対立しすぎないようにすること

\---

## 4-2. 未開放項目の存在感を弱めること

現状の問題:

* STEP1時点で管理ページやデータ項目追加が見えており、存在感が強すぎる

### 修正要件

* 未開放項目は主ボタン表示しない、または弱表示にすること
* 例えば次のいずれかを採用すること

  * 折りたたみ内に格納
  * 「次のSTEPで使えます」セクションにまとめる
  * 文字リンク相当の弱い表示にする
  * サイズを縮小し、主ボタン群と明確に差をつける

目的は:

* 存在は知らせる
* でも今の主役だと誤解させない

\---

## 4-3. ズームUIを常時見やすくすること

現状の問題:

* `zoom-box` が下側に埋もれがち

### 修正要件

* ズームUIは常時見やすい位置に固定表示すること
* 例:

  * canvas 左下固定
  * canvas 右下固定
  * 半透明フローティング

少なくとも、スクロールで見失いにくいこと

\---

## 4-4. Figma系ショートカットを追加すること

ユーザー提案を採用すること。

### 必須ショートカット

* `H` : 手のひらツール（画面移動）
* `V` : 選択ツール
* `C` : 接続ツール
* `Space + Drag` : 一時的な手のひら移動
* `Delete` : 選択中削除
* `Ctrl/Cmd + Z` : Undo
* `Ctrl/Cmd + Y` : Redo

### 補助

* `Ctrl/Cmd + Shift + Z` も Redo として併用対応してよい
* 現在のモードが見える小さなUI表示があるとなお良い
* 注意点として「H/V/C」はキャンバスエリアでの動作とし、入力中には誤動作しないこと

\---

# 5\. サンプル設計の修正（P1）

## 5-1. サンプルは各ステップ単位で開始できるようにすること

現状の問題:

* サンプルから始めると、全部埋まった完成状態になりやすい
* STEP制の意味が薄れている

### 修正要件

サンプルは、**サンプル名 × ステップ** の組み合わせで選べること。

例:

* コーポレートサイト / STEP1から始める
* コーポレートサイト / STEP2から始める
* コーポレートサイト / STEP3から始める
* 記事一覧サイト / STEP1から始める
* 記事一覧サイト / STEP2から始める
* 記事一覧サイト / STEP3から始める

目的:

* STEPごとに「今やるべきこと」に集中できるようにする

\---

# 6\. 実装上の注意事項（抜け漏れ防止）

## 6-1. 再描画時に保持すべきもの

以下を、不必要にリセットしないこと。

* 右パネルのスクロール位置
* canvas のパン位置
* zoom値
* 選択状態
* 接続作業中の仮状態（必要なら）
* 開いているAIタブ

\---

## 6-2. 初回ガイドを追加すること

特に STEP1 白紙開始時は、キャンバス内または左パネル付近に小さくガイドを出すこと。

例:

* まずは「公開ページを追加」から始めます
* ページを追加したら、必要に応じてモーダルや接続線を追加してください
* 右上の「次のSTEPへ」から進めます

\---

## 6-3. ノード初期配置アルゴリズムを見直すこと

ノード追加時に:

* 所属エリア外にはみ出さない
* 既存ノードと極端に重ならない
* STEP1では公開エリア内に収まる
* STEP2では公開 / 管理それぞれ適切なエリア内に置く
* STEP3では DB 項目が DB エリア内に収まる

\---

# 7\. 受け入れ条件（今回の差分で満たすべきこと）

以下を満たしたとき、今回の修正は完了とみなす。

* STEP1 は白紙から始められる
* STEP2 は STEP1完了相当データから始められる
* STEP3 は STEP2完了相当データから始められる
* サンプルがステップ別に開始できる
* STEP1 では ①公開ページエリアが主役サイズになる
* STEP2 では ①公開 + ②管理 が主役サイズになる
* STEP3 では ①②③ が十分使えるサイズになる
* TOPへ戻る / 前のSTEPへ / 次のSTEPへ がわかりやすい
* 保存 / 開く / 外部用に出力 の意味が明確
* 離脱時に未保存警告が出る
* TOPページが狭い幅でも最低限使える
* hero-visual が差し替え可能なヒーロー画像方式になる
* フッターが `©2026 Yobibit` ベースになる
* ファイル名入力欄の拡張子UIが改善される
* CSS一覧 / JS一覧が `|` 区切りではなくなる
* AIタブ切替でスクロールが上に戻らない
* リンク導線一覧が接続線から自動反映される
* 接続線追加モードで移動不能にならない
* 未開放項目の存在感が弱くなる
* ズームUIが常時見つけやすい
* `H / V / C / Space / Ctrl+Y` 系ショートカットが使える

# 8\. Codexへの最終指示文

既存HTML全体を捨てずに活かせる部分は活かしつつ、上記差分を **抜け漏れなく反映** するようにしてください。
既存実装の良い見た目はできるだけ保ちつつ、今回の目的は「完成図を見せること」よりも「初見ユーザーが迷わず作り始められること」を優先することです。差分指示をすべて反映し、既存のindex.htmlを修正してください。
